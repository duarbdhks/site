🚨 운영자는 ClickHouse를 `system` 테이블로 읽습니다

5주차에서는 운영 관측과 장애 판별을 다룹니다. ClickHouse가 느려졌을 때 “서버가 느린 것 같다” 수준에서 멈추면 대응이 어렵습니다. 어떤 쿼리가 느린지, part가 쌓였는지, merge가 밀리는지, mutation이 적체됐는지, 메모리가 어디서 터지는지 구분해야 합니다.

ClickHouse는 내부 상태를 `system` database의 여러 테이블로 제공합니다. 예를 들어 `system.query_log`는 실행된 쿼리 이력을 보고, `system.parts`는 테이블의 part 상태를 보고, `system.merges`는 진행 중인 merge를 보고, `system.mutations`는 ALTER/DELETE/UPDATE 계열 mutation 진행 상태를 봅니다. 이 테이블들은 운영자의 계기판입니다.

핵심 문장
ClickHouse 장애 대응은 “느리다”를 “어떤 내부 큐와 리소스가 병목인가”로 번역하는 작업입니다. system 테이블은 그 번역을 위한 공통 언어입니다.

🧭 1강. 장애를 5가지 범주로 먼저 나눕니다

ClickHouse 장애나 성능 저하는 크게 다섯 가지로 나눠 볼 수 있습니다. 첫째, 쿼리 자체가 무겁습니다. 둘째, insert가 너무 잘게 들어와 part가 폭증합니다. 셋째, background merge가 밀립니다. 넷째, mutation이 오래 걸립니다. 다섯째, 외부 파이프라인 Kafka/Flink 쪽 지연이 ClickHouse 문제처럼 보입니다.

이 분류가 중요한 이유는 대응 방식이 다르기 때문입니다. 무거운 쿼리는 쿼리와 테이블 설계를 봐야 합니다. part 폭증은 적재 batch와 insert 빈도를 봐야 합니다. merge 지연은 디스크, CPU, part 수, 테이블 설계를 봐야 합니다. mutation 적체는 DELETE/UPDATE/ALTER 사용 방식을 봐야 합니다. Kafka/Flink 지연은 ClickHouse가 원인이 아닐 수도 있습니다.

증상먼저 의심할 것볼 system 테이블대표 대응
특정 API만 느림쿼리 설계, `FINAL`, JOIN`system.query_log`, `system.processes`쿼리 수정, 집계 테이블, 범위 제한
전체적으로 insert 후 느림작은 insert, part 폭증`system.parts`, `system.part_log`batch 크기 조정, async insert 검토
디스크/CPU 지속 상승merge backlog`system.merges`, `system.parts`part 수 감소, 파티션 설계 점검

📜 2강. `system.query_log`는 느린 쿼리의 블랙박스입니다

`system.query_log`는 실행된 쿼리의 시간, 읽은 행과 바이트, 메모리 사용량, 예외, 사용자, 데이터베이스, 테이블 같은 정보를 담습니다. API 장애가 발생했을 때 가장 먼저 확인할 후보입니다. 단순히 “어떤 쿼리가 느렸나”뿐 아니라 “느린 쿼리가 얼마나 많은 데이터를 읽었나”를 볼 수 있습니다.

느린 쿼리를 볼 때는 duration만 보면 부족합니다. 읽은 rows/bytes가 큰 쿼리인지, memory_usage가 높은 쿼리인지, exception이 반복되는지, 특정 사용자나 API에서만 발생하는지 함께 봐야 합니다. 예를 들어 10초 걸렸지만 읽은 데이터가 작다면 lock, network, external dictionary, 동시성 문제일 수 있습니다. 반대로 1초지만 매초 100번 실행되면 전체 부하의 원인일 수 있습니다.

`query_log`는 사후 분석에 특히 유용합니다. 장애가 끝난 뒤에도 어떤 시간대에 어떤 쿼리가 급증했는지 볼 수 있기 때문입니다. 규완님이 운영 문서를 만든다면 “최근 1시간의 상위 메모리 쿼리”, “최근 1시간의 상위 read_bytes 쿼리”, “exception이 많은 쿼리” 정도는 표준 쿼리로 갖춰두는 것이 좋습니다.

운영 감각
느린 쿼리는 duration, read_rows, read_bytes, memory_usage, query kind를 함께 봐야 합니다. 단일 숫자 하나로 원인을 단정하면 오진 가능성이 큽니다.

🧱 3강. `system.parts`는 테이블 건강검진표입니다

`system.parts`는 MergeTree 테이블의 part 상태를 보여줍니다. part는 ClickHouse가 데이터를 저장하는 기본 조각입니다. part가 너무 많으면 쿼리와 merge 모두 부담이 커질 수 있습니다. 특히 작은 insert가 빈번하면 작은 part가 많이 생기고, 백그라운드 merge가 이를 따라가지 못할 수 있습니다.

part 상태를 볼 때는 active part 수, 파티션별 part 수, rows, bytes, modification time을 봅니다. 특정 파티션에 active part가 과도하게 많다면 해당 기간에 insert가 잘게 들어왔거나 merge가 밀리고 있을 가능성이 있습니다. 파티션이 너무 잘게 쪼개져 있으면 각 파티션마다 part가 쌓여 관리 비용이 커집니다.

예를 들어 일별 파티션에 테넌트별로 작은 insert가 초당 수십 번 들어오면 하루 동안 수많은 part가 만들어질 수 있습니다. ClickHouse는 merge로 정리하려 하지만, merge도 CPU와 디스크 I/O를 씁니다. merge가 밀리면 새 쿼리는 더 많은 part를 읽어야 하고, 다시 느려집니다. 이것이 part 폭증의 악순환입니다.

작은 insert가 많은 경우 초당 작은 batch → 작은 part 다수 생성 → merge backlog 증가 → 쿼리 read overhead 증가 → 전체 지연 큰 batch로 모으는 경우 적절한 batch → part 수 감소 → merge 부담 감소 → 조회 안정성 증가

⚙️ 4강. `system.merges`는 백그라운드 정리 작업을 보여줍니다

`system.merges`는 현재 진행 중인 merge와 part mutation 정보를 보여줍니다. merge는 ClickHouse가 작은 part를 더 큰 part로 합치고, 엔진별 정리 규칙을 적용하는 백그라운드 작업입니다. `ReplacingMergeTree`의 중복 정리도 merge에 의존합니다. 따라서 merge 상태는 성능뿐 아니라 정합성 체감에도 영향을 줍니다.

merge가 계속 밀려 있다면 몇 가지 원인을 볼 수 있습니다. insert가 너무 잘게 들어오는지, 파티션이 너무 많은지, 디스크 I/O가 부족한지, 동시에 무거운 쿼리가 많은지, mutation이 merge 리소스를 잡아먹는지 확인해야 합니다. 단순히 `OPTIMIZE FINAL`을 반복 실행하는 것은 근본 해결이 아닐 수 있습니다.

merge는 정상적인 백그라운드 작업입니다. merge가 보인다고 바로 문제는 아닙니다. 문제는 merge가 입력 속도를 따라가지 못하고 backlog처럼 쌓이는 상황입니다. 운영자는 “merge가 있다/없다”가 아니라 “현재 insert와 part 증가 속도 대비 merge가 따라오는가”를 봐야 합니다.

🧬 5강. `system.mutations`는 무거운 변경 작업의 진행표입니다

ClickHouse에서 `ALTER TABLE ... DELETE`, `ALTER TABLE ... UPDATE` 같은 작업은 mutation으로 처리됩니다. Mutation은 기존 데이터를 즉시 제자리에서 바꾸는 것이 아니라, 관련 part를 다시 쓰는 방식으로 진행됩니다. 데이터량이 크면 오래 걸리고, 리소스를 많이 사용할 수 있습니다.

`system.mutations`는 mutation 명령, 생성 시간, 완료 여부, 남은 part 수, 실패 메시지 등을 확인하는 데 사용합니다. 삭제나 보정 작업을 실행했는데 끝나지 않거나, 쿼리 성능이 갑자기 나빠졌다면 mutation 적체를 확인해야 합니다. 특히 운영 중 대량 DELETE를 자주 실행하는 설계는 ClickHouse와 잘 맞지 않을 수 있습니다.

가능하면 삭제는 파티션 단위로 처리하는 것이 더 안전합니다. 예를 들어 월별 보존 정책이면 오래된 월 파티션을 통째로 정리할 수 있습니다. 특정 고객 데이터 삭제 같은 요구는 테이블 설계 단계에서 별도 전략이 필요합니다. row-level delete를 자주 쓰는 것은 마지막 수단에 가깝습니다.

🚦 6강. `system.processes`는 지금 실행 중인 쿼리를 보여줍니다

`system.processes`는 현재 실행 중인 쿼리를 확인할 때 사용합니다. 장애가 발생한 바로 그 순간에는 `query_log`보다 `processes`가 더 직접적일 수 있습니다. 어떤 쿼리가 지금 메모리를 쓰고 있는지, 얼마나 오래 실행 중인지, 어떤 사용자가 실행했는지 볼 수 있습니다.

운영 중에는 “지금 당장 죽여야 할 쿼리”와 “기다려도 되는 쿼리”를 구분해야 합니다. 관리자 콘솔의 실수성 전체 기간 쿼리, API에서 무한 범위를 읽는 쿼리, `FINAL + JOIN`이 붙은 대량 쿼리는 중단 후보가 될 수 있습니다. 반면 백필이나 정기 리포트 쿼리는 의도된 작업일 수 있습니다.

쿼리를 kill하는 것은 증상 완화이지 원인 해결이 아닙니다. 반드시 이후에 query_log를 보고 누가 어떤 경로로 실행했는지, API 기본값이 잘못됐는지, 권한과 리소스 그룹이 필요한지, 쿼리 제한 설정이 필요한지 후속 조치를 해야 합니다.

📈 7강. ClickHouse 문제와 Kafka/Flink 문제를 분리하기

대시보드 데이터가 늦게 보이면 사용자는 “ClickHouse가 느리다”고 말할 수 있습니다. 하지만 실제 원인은 Kafka lag, Flink backpressure, CDC connector 지연, ClickHouse insert 지연, MV 반영 지연 중 하나일 수 있습니다. 운영자는 파이프라인 단계를 나눠 지연 위치를 찾아야 합니다.

가장 쉬운 방법은 각 단계의 timestamp를 남기는 것입니다. 원천 이벤트 발생 시각, CDC 캡처 시각, Kafka publish 시각, Flink 처리 시각, ClickHouse insert 시각, API 조회 시각을 비교할 수 있으면 병목 위치가 보입니다. 예를 들어 원천 이벤트와 Kafka publish 사이가 벌어지면 CDC 문제이고, Kafka와 ClickHouse insert 사이가 벌어지면 Flink 또는 sink 문제일 수 있습니다.

ClickHouse 내부만 봐서는 upstream 지연을 확정하기 어렵습니다. 그러나 ClickHouse 테이블에 `ingested_at`, `source_updated_at`, `event_time` 같은 컬럼이 있으면 데이터 자체로 지연을 추정할 수 있습니다. 이런 컬럼은 단순 메타데이터가 아니라 장애 분석용 센서입니다.

event_time → captured_at → kafka_published_at → flink_processed_at → clickhouse_inserted_at → api_queried_at 각 timestamp 차이를 보면 지연 위치를 좁힐 수 있습니다.

🧯 8강. 운영 런북은 “증상 → 확인 → 판단 → 조치” 순서여야 합니다

런북(runbook, 장애 대응 절차서)은 SQL 모음만으로는 부족합니다. 운영자는 장애 상황에서 무엇을 먼저 보고, 어떤 결과면 어떤 판단을 하고, 어떤 조치를 할지 알아야 합니다. 좋은 런북은 사람의 판단 시간을 줄입니다.

예를 들어 “API 응답이 느림”이라는 증상에 대해 먼저 `system.processes`로 현재 무거운 쿼리를 보고, `query_log`로 최근 read_bytes 상위 쿼리를 확인하고, 해당 쿼리에 `FINAL`, JOIN, 넓은 날짜 범위가 있는지 판단합니다. 동시에 `system.parts`에서 대상 테이블 part 수가 비정상인지 봅니다. 이렇게 단계가 있어야 오진을 줄입니다.

증상확인판단조치
특정 API 느림`query_log`, `processes`쿼리 read_bytes와 memory_usage 확인쿼리 제한, 집계 테이블 전환, 캐시 검토
insert 지연`parts`, 파이프라인 lag작은 insert/merge backlog 여부batch 조정, sink 병렬도 조정
삭제 작업 지연`mutations`남은 part와 실패 메시지 확인mutation 중단/재설계, 파티션 단위 처리
데이터 숫자 틀림raw/fact/agg/serving 비교어느 계층에서 차이 발생했는지 확인해당 기간 재처리, MV target 보정

📣 9강. 매니저/CTO 관점의 ClickHouse 운영 언어

규완님이 기술 리더 역할을 의식한다면, ClickHouse 운영 이슈를 팀과 비즈니스 언어로 번역할 수 있어야 합니다. “merge가 밀렸습니다”에서 끝나면 비개발자는 이해하기 어렵습니다. “데이터 조각이 너무 잘게 쌓여 조회 시 읽을 조각이 늘었고, 백그라운드 정리 속도가 유입 속도를 따라가지 못했습니다”라고 설명하면 원인과 조치가 보입니다.

정합성 이슈도 마찬가지입니다. “ReplacingMergeTree라 아직 FINAL이 필요합니다”보다 “동일 주문의 여러 버전이 아직 정리되기 전이라, 정확한 현재 상태를 보려면 조회 시점 보정이 필요하고 그만큼 응답 비용이 증가합니다”라고 말하는 편이 의사결정에 도움이 됩니다.

운영 보고는 항상 영향, 원인, 조치, 재발 방지로 정리합니다. 영향은 어떤 API와 고객이 영향을 받았는지입니다. 원인은 쿼리, part, merge, mutation, upstream 중 어디였는지입니다. 조치는 즉시 완화와 구조 개선을 나눕니다. 재발 방지는 쿼리 제한, batch 조정, 테이블 재설계, 모니터링 알림 같은 항목입니다.

📡 10강. 알림은 “증상 알림”과 “원인 후보 알림”을 나눠야 합니다

모니터링 알림은 많을수록 좋은 것이 아닙니다. 알림이 너무 많으면 팀은 알림을 무시하게 됩니다. ClickHouse에서는 사용자 영향이 있는 증상 알림과, 앞으로 문제가 될 가능성이 있는 원인 후보 알림을 나눠 설계하는 것이 좋습니다. 예를 들어 API p95 지연 상승은 증상 알림이고, 특정 테이블의 active part 수 급증은 원인 후보 알림입니다.

증상 알림은 즉시 대응이 필요합니다. API 오류율, 응답 지연, 대시보드 freshness 지연, insert 실패율 같은 항목입니다. 원인 후보 알림은 조기 경고에 가깝습니다. part 수 증가, merge backlog, mutation 적체, 디스크 사용률 상승, 특정 쿼리 read_bytes 급증 같은 항목입니다.

좋은 알림은 actionability(조치 가능성)가 있습니다. “ClickHouse warning” 같은 알림은 쓸모가 낮습니다. “최근 15분 동안 `agg_daily_campaign`의 active part 수가 평소 대비 3배 이상 증가했고, merge가 따라오지 못합니다”처럼 조치 방향을 좁혀주는 알림이 좋습니다. 임계값은 서비스 규모와 평소 패턴에 맞춰 조정해야 하므로 처음에는 보수적으로 시작해 운영 데이터로 다듬습니다.

🧭 11강. Freshness는 통계 시스템의 사용자 체감 지표입니다

ClickHouse 자체가 빠르게 응답해도 데이터가 오래된 상태라면 사용자는 장애로 느낍니다. freshness(신선도)는 “대시보드에 보이는 데이터가 원천 이벤트 대비 얼마나 최신인가”를 나타냅니다. 통계 시스템에서는 쿼리 latency만큼 freshness가 중요합니다.

freshness를 측정하려면 데이터에 시간 표식이 필요합니다. `event_time`, `source_updated_at`, `ingested_at`, `aggregated_at` 같은 컬럼이 있으면 현재 시각과 비교해 지연을 계산할 수 있습니다. 예를 들어 최신 `event_time`이 현재보다 20분 전이라면 upstream 지연일 수 있고, 최신 `ingested_at`은 현재인데 `aggregated_at`이 오래됐다면 MV 또는 집계 단계 문제일 수 있습니다.

freshness SLA를 지표별로 정하면 운영 판단이 쉬워집니다. 실시간 운영 대시보드는 1분 이내, 고객 리포트는 1시간 이내, 정산 리포트는 D+1 마감처럼 구분할 수 있습니다. 모든 지표에 같은 freshness를 요구하면 비용이 증가합니다.

지표 유형Freshness 예시운영 의미
실시간 모니터링1~5분지연이 짧아야 하지만 일부 변동 허용
고객 대시보드5~30분안정성과 비용 균형

🧪 12강. 장애 복기에서는 “재현 가능한 증거”를 남깁니다

운영 장애가 끝난 뒤 가장 아쉬운 상황은 증거가 남아 있지 않은 경우입니다. 누가 어떤 쿼리를 실행했는지, 당시 part 수가 어땠는지, merge가 밀렸는지, mutation이 있었는지 기록이 없으면 재발 방지책이 추정에 머뭅니다. `system` 테이블과 외부 모니터링을 함께 사용해 당시 상태를 보존해야 합니다.

복기 문서에는 타임라인이 중요합니다. 언제 지연이 시작됐고, 언제 알림이 울렸고, 어떤 쿼리나 적재 작업이 증가했고, 어떤 조치를 했고, 언제 회복됐는지 순서대로 적습니다. 그 다음 원인을 하나로 단정하기보다 증거와 추론을 분리합니다. 예를 들어 “12:05부터 read_bytes 상위 쿼리가 증가했고, 같은 시점 API p95가 상승했다”는 증거입니다. “이 쿼리가 주 원인으로 추정된다”는 판단입니다.

복기의 목적은 책임 추궁이 아니라 시스템 개선입니다. 쿼리 제한이 없었는지, 백필이 피크 시간에 실행됐는지, 알림이 늦었는지, 런북이 부족했는지 확인해야 합니다. ClickHouse 운영 성숙도는 장애가 없느냐보다 같은 장애가 반복되지 않느냐로 판단하는 편이 현실적입니다.

🏢 13강. Cloud와 자체 운영에서 system 테이블 해석은 달라질 수 있습니다

ClickHouse Cloud와 자체 운영은 관측 방식이 일부 다를 수 있습니다. Cloud에서는 일부 system 테이블이 노드별 로컬 정보를 담고 있어 전체 클러스터 관점을 보려면 별도 함수나 Cloud 콘솔 지표를 함께 봐야 할 수 있습니다. 자체 운영에서는 서버, 디스크, 네트워크, Keeper, replica 상태까지 직접 책임져야 합니다.

따라서 런북에는 “우리 환경에서 이 쿼리가 전체를 보는가, 현재 노드만 보는가”가 명시되어야 합니다. 클러스터 환경에서 한 노드만 보고 정상이라고 판단하면 다른 replica의 문제를 놓칠 수 있습니다. 반대로 모든 노드를 매번 무겁게 조회하면 진단 쿼리 자체가 부담이 될 수 있습니다.

규완님이 팀 문서를 만든다면 Cloud/OSS, 단일 노드/클러스터, shard/replica, 권한 범위를 첫 페이지에 적는 것이 좋습니다. 운영 런북은 환경 의존성이 강하므로 일반 문서를 그대로 복사하면 빈틈이 생깁니다.

🧰 14강. 운영 대시보드는 적은 수의 강한 신호로 시작합니다

ClickHouse 운영 대시보드를 만들 때 처음부터 수십 개 그래프를 넣으면 오히려 보기 어렵습니다. 초반에는 강한 신호를 중심으로 구성하는 편이 좋습니다. API latency, query read bytes, memory usage 상위 쿼리, active part 수, merge 진행, mutation backlog, insert 실패율, data freshness 정도면 많은 장애를 1차 분류할 수 있습니다.

각 그래프는 질문과 연결되어야 합니다. active part 수 그래프는 “작은 insert가 쌓이고 있는가”를 묻습니다. read bytes 상위 쿼리는 “어떤 쿼리가 리소스를 쓰는가”를 묻습니다. freshness 그래프는 “데이터가 늦게 들어오는가, 집계가 늦는가”를 묻습니다. 질문 없는 그래프는 장애 대응에서 시선을 분산시킵니다.

운영 성숙도가 올라가면 워크로드별로 대시보드를 나눌 수 있습니다. 사용자 API용, 백필/배치용, 적재 파이프라인용, 클러스터 리소스용을 분리하면 담당자가 빠르게 볼 수 있습니다. 하지만 시작점은 항상 단순해야 합니다. 한 화면에서 현재 장애 범주를 1분 안에 좁힐 수 있으면 좋은 운영 대시보드입니다.

✅ 5주차 정리

이번 주의 핵심은 ClickHouse 운영을 `system` 테이블 중심으로 읽는 것입니다. `system.query_log`는 느린 쿼리와 리소스 사용을 보고, `system.parts`는 테이블 part 건강 상태를 보고, `system.merges`는 백그라운드 병합 작업을 보고, `system.mutations`는 무거운 변경 작업의 진행을 봅니다.

장애는 “ClickHouse가 느림”으로 뭉뚱그리지 말고 쿼리, part 폭증, merge 지연, mutation 적체, upstream lag로 나눠야 합니다. 각 단계의 timestamp와 system 테이블을 함께 보면 원인을 좁힐 수 있습니다. 런북은 증상에서 시작해 확인, 판단, 조치로 이어져야 합니다.

핵심 용어

`query_log`는 실행된 쿼리 이력입니다. `parts`는 MergeTree 테이블의 데이터 조각 상태입니다. `merges`는 진행 중인 백그라운드 병합 작업입니다. `mutations`는 대량 UPDATE/DELETE/ALTER 같은 변경 작업의 진행 정보입니다.
