📥 ClickHouse 운영의 마지막 축은 “어떻게 넣고, 다시 넣고, 오래 둘 것인가”입니다

6주차에서는 적재, 재처리, 비용, 보존 정책을 함께 다룹니다. ClickHouse 성능은 조회만으로 결정되지 않습니다. 데이터를 너무 작게 자주 넣으면 part가 늘고, 중복 재시도를 고려하지 않으면 숫자가 틀리고, 백필 절차가 없으면 장애 복구가 위험해지고, 보존 정책이 없으면 비용이 계속 증가합니다.

규완님 환경에서는 `CDC → Kafka → Flink → ClickHouse` 흐름이 핵심입니다. 이 흐름에서 ClickHouse는 마지막 저장소처럼 보이지만, 실제로는 파이프라인 계약의 결과를 받는 지점입니다. upstream이 어떤 보장으로 데이터를 보내는지, ClickHouse가 어떤 방식으로 받아들이는지, 실패 시 어디서부터 다시 처리할지 정해야 합니다.

핵심 문장
ClickHouse 적재 설계는 insert 성능 문제가 아니라 정합성, 재처리, 비용, 운영 복구를 함께 결정하는 시스템 설계입니다.

🧺 1강. Batch insert는 성능 최적화가 아니라 기본 위생입니다

ClickHouse는 대량 insert에 강하지만, 아주 작은 insert를 매우 자주 보내는 방식에는 취약할 수 있습니다. insert마다 part가 생기고, 많은 작은 part는 이후 merge 부담을 만듭니다. 따라서 데이터를 적절한 batch(묶음)로 모아 넣는 것은 성능 튜닝 이전의 기본 위생에 가깝습니다.

예를 들어 이벤트 한 건마다 ClickHouse에 insert하면 초당 수천 개의 작은 part가 생길 수 있습니다. 반대로 Kafka/Flink sink에서 일정 건수 또는 일정 시간 단위로 모아 insert하면 part 수가 줄고 압축과 merge 효율이 좋아집니다. 중요한 것은 지연 허용 범위와 part 관리 사이의 균형입니다.

실시간 대시보드라고 해서 반드시 1건씩 즉시 넣어야 하는 것은 아닙니다. 사용자가 체감하는 실시간이 1초인지, 10초인지, 1분인지 정의해야 합니다. 10초 지연을 허용할 수 있다면 batch 효율을 얻을 수 있습니다. 지연 허용 범위를 명확히 하지 않으면 시스템은 비용이 큰 방향으로 흐르기 쉽습니다.

🔌 2강. Kafka에서 ClickHouse로 넣는 대표 방식

Kafka 데이터를 ClickHouse로 넣는 방식은 여러 가지가 있습니다. 대표적으로 Flink sink, Kafka Connect Sink, ClickHouse Kafka table engine, ClickPipes 같은 관리형 경로를 생각할 수 있습니다. 어떤 방식이 맞는지는 이미 운영 중인 인프라, exactly-once 요구 수준, 변환 복잡도, 장애 복구 방식에 따라 달라집니다.

Flink sink는 규완님 환경에서 자연스러운 선택입니다. 이미 Flink에서 변환, window, dedup, enrichment(외부 정보 보강)를 하고 있다면 ClickHouse는 정제된 결과를 받으면 됩니다. Flink의 checkpoint와 sink semantics를 맞추면 재시도와 중복 관리 전략을 비교적 명확히 가져갈 수 있습니다.

Kafka Connect Sink는 Kafka 중심 조직에서 운영하기 쉽습니다. 별도 스트림 처리 로직이 복잡하지 않고, Kafka topic을 ClickHouse table로 안정적으로 밀어 넣는 것이 목표라면 후보가 됩니다. Kafka table engine은 ClickHouse 내부에서 Kafka를 읽는 방식으로 단순해 보이지만, 운영 책임과 확장성, 장애 복구 모델을 이해해야 합니다.

방식장점주의점규완님 환경에서의 판단
Flink Sink변환, dedup, window 처리와 결합하기 좋습니다.Flink 상태와 checkpoint 운영 복잡도가 있습니다.이미 Flink가 있다면 1순위 후보입니다.
Kafka Connect SinkKafka 기반 적재를 표준화하기 쉽습니다.복잡한 비즈니스 보정은 외부에서 해야 합니다.단순 topic-to-table 적재에 적합합니다.
Kafka table engineClickHouse 내부에서 Kafka를 직접 읽습니다.소비/재처리/확장 책임을 명확히 해야 합니다.단순 구조에는 좋지만 운영 모델 검토가 필요합니다.
관리형 ClickPipes운영 부담을 줄일 수 있습니다.Cloud 환경과 비용, 지원 범위를 확인해야 합니다.ClickHouse Cloud 사용 시 검토 가치가 있습니다.

🔁 3강. Retry는 중복을 만든다는 전제로 설계합니다

분산 시스템에서 retry는 필요합니다. 네트워크가 끊기거나, ClickHouse가 일시적으로 바쁘거나, 클라이언트가 응답을 받지 못할 수 있습니다. 문제는 retry가 중복 insert를 만들 수 있다는 점입니다. 요청이 실제로는 성공했는데 클라이언트가 실패로 판단하고 다시 보내면 같은 batch가 두 번 들어갑니다.

따라서 적재 설계에서는 idempotency(멱등성, 같은 작업을 여러 번 실행해도 결과가 같게 만드는 성질)를 고려해야 합니다. append-only 이벤트라면 event_id 기반 중복 제거가 필요할 수 있습니다. 집계 delta라면 같은 delta가 두 번 적용되지 않도록 batch_id나 source offset을 관리해야 합니다. 최신 상태 테이블이라면 version 기준으로 오래된 이벤트가 최신을 덮지 않게 해야 합니다.

ClickHouse에는 insert retry 중복을 줄이기 위한 설정과 전략이 있지만, 모든 상황을 자동으로 해결한다고 보면 안 됩니다. 특히 materialized view가 붙은 insert, 여러 테이블에 걸친 적재, Flink checkpoint와 ClickHouse commit 경계는 별도 검토가 필요합니다. “재시도하면 중복될 수 있다”는 전제에서 downstream이 안전해야 합니다.

운영 기준
성공 여부를 애매하게 만든 실패가 가장 위험합니다. ClickHouse에 들어갔는지 모르겠는 batch는 재시도될 수 있고, 그 결과 중복이 생길 수 있습니다. batch_id, source offset, event_id를 추적 가능하게 두세요.

🧯 4강. Backfill은 “과거 데이터를 다시 넣기”가 아니라 “기존 결과를 교체하기”입니다

Backfill(백필, 과거 데이터 재처리)은 통계 시스템에서 필수입니다. 버그가 난 기간을 다시 계산하거나, 새 지표를 과거에도 채우거나, CDC 누락 구간을 복구해야 할 수 있습니다. 하지만 ClickHouse에서 백필을 단순히 같은 데이터를 다시 insert하는 것으로 보면 중복 집계가 생길 수 있습니다.

안전한 백필은 보통 staging table을 사용합니다. 먼저 과거 기간 데이터를 staging에 다시 만들고, 검증한 뒤 target의 해당 기간 파티션 또는 범위를 교체합니다. 이렇게 하면 사용자에게 중간 상태가 보이는 시간을 줄이고, 문제가 생기면 기존 데이터를 유지할 수 있습니다.

집계 테이블의 백필은 특히 조심해야 합니다. raw 데이터를 다시 넣으면 MV가 다시 실행되어 target 집계가 두 배가 될 수 있습니다. 그래서 백필 경로와 실시간 경로를 분리하거나, target 기간을 지우고 다시 채우거나, 새 테이블을 만들어 검증 후 swap하는 방식이 필요합니다.

위험한 백필 과거 raw 재삽입 → MV 재실행 → 기존 agg에 중복 누적 안전한 백필 예시 과거 데이터 재계산 → staging_agg 생성 → 검증 → target 기간 교체

🗃️ 5강. TTL과 보존 정책은 비용 제어 장치입니다

ClickHouse는 데이터를 오래 저장할수록 디스크 비용과 백그라운드 관리 비용이 증가합니다. TTL(Time To Live, 데이터 수명 정책)은 일정 시간이 지난 데이터를 삭제하거나 이동하는 정책입니다. 예를 들어 raw 이벤트는 90일, fact는 1년, daily aggregate는 3년 보존처럼 계층별 보존 기간을 다르게 둘 수 있습니다.

모든 데이터를 영구 보존하려는 생각은 비용과 운영 리스크를 키웁니다. raw 이벤트는 재처리에 유용하지만 가장 큽니다. 집계 테이블은 작지만 질문 형태가 제한됩니다. 따라서 원본 보존 기간과 집계 보존 기간을 분리해야 합니다. “무엇을 다시 계산할 수 있어야 하는가”와 “얼마나 오래 상세 추적이 필요한가”가 기준입니다.

보존 정책은 기술 결정이면서 비즈니스 결정입니다. 정산, 법적 감사, 고객 계약, 개인정보 삭제 요구, 비용 예산이 함께 얽힙니다. 개발팀 단독으로 “90일이면 충분”이라고 정하기보다, 지표별 사용처와 리스크를 확인해야 합니다.

계층보존 예시이유주의점
`raw_*`30~180일재처리와 원인 추적에 필요합니다.가장 크므로 비용 영향이 큽니다.
`fact_*`6개월~2년분석 기준으로 정제되어 재사용성이 높습니다.정제 로직 변경 시 재생성 가능성이 필요합니다.
`agg_*`1~5년작고 리포트에 유용합니다.질문 형태가 제한됩니다.
`serving_*`API 요구 기준응답 속도와 SLA에 직접 연결됩니다.원본과 불일치 시 재생성 절차가 필요합니다.

🧊 6강. Hot data와 cold data를 나누면 비용 구조가 보입니다

모든 데이터가 같은 빈도로 조회되지는 않습니다. 최근 7일 데이터는 대시보드에서 자주 읽히고, 최근 90일은 리포트에서 가끔 읽히며, 1년 전 raw 이벤트는 장애 분석이나 감사 때만 필요할 수 있습니다. 이런 차이를 hot data(자주 읽는 데이터)와 cold data(드물게 읽는 데이터)로 나눌 수 있습니다.

Hot data는 빠른 디스크와 적극적인 집계, 캐시, serving 테이블이 중요합니다. Cold data는 비용 효율적인 저장과 느린 조회 허용이 중요합니다. ClickHouse Cloud나 자체 운영 환경에서 스토리지 계층을 어떻게 나눌지는 환경별 확인이 필요하지만, 사고방식은 같습니다. 자주 읽는 데이터에 비용을 집중하고, 드물게 읽는 데이터는 저렴하게 보관합니다.

이 구분은 API 설계에도 영향을 줍니다. 최근 7일 대시보드는 빠른 응답을 보장하고, 1년치 상세 raw 다운로드는 비동기 리포트로 분리할 수 있습니다. 모든 요청을 같은 SLA로 처리하려 하면 시스템 비용이 높아집니다.

🧮 7강. 압축과 컬럼 타입은 장기 비용에 영향을 줍니다

ClickHouse는 컬럼 지향 저장과 압축 효율이 강점입니다. 하지만 컬럼 타입을 대충 잡으면 저장 비용과 쿼리 비용이 증가할 수 있습니다. 예를 들어 숫자로 표현할 수 있는 상태 값을 긴 문자열로 저장하거나, 낮은 cardinality의 문자열을 아무 처리 없이 대량 저장하면 비효율이 생길 수 있습니다.

시간 컬럼도 중요합니다. `Date`, `DateTime`, `DateTime64` 중 어떤 정밀도가 필요한지 결정해야 합니다. 모든 이벤트에 밀리초나 나노초 정밀도가 필요한 것은 아닙니다. 정밀도는 분석 요구에 맞춰 선택해야 합니다. 금액은 부동소수점보다 정수 단위 또는 Decimal을 검토하는 편이 안전합니다.

압축 codec은 고급 주제처럼 보이지만, 데이터량이 커지면 비용에 직접 영향을 줍니다. 다만 처음부터 모든 컬럼에 codec을 세밀하게 지정하기보다, 큰 테이블의 큰 컬럼부터 관찰해 조정하는 방식이 현실적입니다. 기본 설정으로 시작하되, 저장 비용 상위 컬럼을 찾아 개선하는 순서가 낫습니다.

🧭 8강. 재처리 가능한 구조가 장애 복구 시간을 줄입니다

통계 시스템은 결국 언젠가 틀립니다. CDC 누락, Flink 버그, 지표 정의 변경, 중복 이벤트, 늦은 삭제가 발생합니다. 중요한 것은 틀리지 않는 시스템을 상상하는 것이 아니라, 틀렸을 때 어느 구간을 안전하게 다시 만들 수 있는 구조를 갖추는 것입니다.

재처리 가능성을 높이려면 원본 또는 충분히 원본에 가까운 데이터를 일정 기간 보존해야 합니다. 또한 각 계층의 산출물이 deterministic(결정적, 같은 입력이면 같은 출력)이어야 합니다. 같은 기간을 다시 계산했을 때 결과가 달라진다면 로직, 외부 참조 데이터, 시간 기준 중 무엇이 달라졌는지 추적해야 합니다.

재처리 단위는 보통 날짜, 파티션, 테넌트, 지표 단위로 잡습니다. 너무 큰 단위는 비용이 크고, 너무 작은 단위는 절차가 복잡합니다. 1주차의 파티션 설계가 6주차의 백필 난이도와 연결되는 이유입니다.

📌 9강. 규완님에게 맞는 최종 설계 원칙

첫째, 적재는 batch 단위로 안정화합니다. Flink에서 ClickHouse로 넣는다면 batch 크기, flush interval, checkpoint, retry 정책을 함께 봐야 합니다. 둘째, 중복 가능성을 전제로 event_id, batch_id, source offset, version을 남깁니다. 셋째, API가 보는 serving 테이블과 raw/fact 계층을 분리합니다.

넷째, 백필은 staging과 교체 절차를 기본으로 합니다. 과거 데이터를 그냥 다시 insert하는 것은 위험합니다. 다섯째, 보존 정책은 계층별로 다르게 둡니다. raw는 재처리 가능 기간만큼, agg는 리포트 기간만큼, serving은 API 요구만큼 둡니다. 여섯째, 비용 최적화는 데이터 사용 빈도와 컬럼 크기를 관찰한 뒤 진행합니다.

이 원칙은 기술적으로도 중요하지만 팀 운영에도 중요합니다. 장애가 났을 때 “어떤 기간의 어떤 계층을 다시 만들면 되는가”가 명확하면 복구 시간이 줄어듭니다. 반대로 모든 데이터가 한 테이블에 섞여 있고 백필 절차가 없다면, 작은 지표 버그도 큰 운영 리스크가 됩니다.

⚡ 10강. Async insert는 지연과 안정성의 교환 관계를 이해하고 써야 합니다

Async insert(비동기 적재)는 클라이언트가 보낸 insert를 ClickHouse가 내부에서 모아 처리하도록 도와 part 생성을 줄이는 데 활용될 수 있습니다. 작은 insert가 잦은 환경에서는 매력적입니다. 하지만 비동기라는 말은 성공 응답 시점, 버퍼링, 실패 처리, 중복 가능성을 더 꼼꼼히 봐야 한다는 뜻이기도 합니다.

비동기 적재는 “작은 insert를 모아 효율을 높인다”는 측면에서 좋지만, 파이프라인이 요구하는 전달 보장과 맞아야 합니다. 사용자가 insert 응답을 받았을 때 데이터가 즉시 조회되어야 하는지, 일정 시간 뒤 보여도 되는지, 실패 시 재시도를 어떻게 할지 정해야 합니다. 실시간성이 강한 API와 배치성 로그 적재는 요구가 다릅니다.

규완님 환경에서 Flink가 이미 batch를 만들어 넣고 있다면 async insert가 반드시 필요한 것은 아닐 수 있습니다. 반대로 여러 작은 producer가 직접 ClickHouse에 쓰는 구조라면 도움이 될 수 있습니다. 즉, async insert는 “항상 켜는 옵션”이 아니라 현재 적재 패턴의 part 생성 문제를 보고 선택하는 카드입니다.

🧬 11강. 스키마 변경은 데이터 파이프라인 전체 이벤트입니다

ClickHouse 테이블에 컬럼 하나를 추가하는 일은 단순 DDL처럼 보이지만, CDC, Kafka schema, Flink job, MV, API, 리포트가 모두 연결되어 있으면 전체 파이프라인 이벤트가 됩니다. 원천 DB에 컬럼이 추가됐는데 ClickHouse 테이블과 MV target에 반영되지 않으면 데이터가 누락되거나 job이 실패할 수 있습니다.

스키마 변경을 안전하게 하려면 backward compatibility(하위 호환성)를 고려합니다. 새 컬럼은 nullable 또는 default를 두고 추가하고, producer가 새 값을 보내기 전 consumer가 먼저 받을 준비를 하는 방식이 안전합니다. MV가 source의 `SELECT *`에 의존하면 컬럼 추가 때 예상치 못한 영향이 생길 수 있으므로 명시적 컬럼 목록을 선호하는 편이 좋습니다.

집계 테이블의 스키마 변경은 더 어렵습니다. 새 지표를 추가하면 과거 기간 값이 비어 있을 수 있습니다. API는 과거 값을 0으로 볼지, null로 볼지, “지원 시작일 이후만 제공”으로 볼지 정해야 합니다. 스키마 변경은 기술 작업이면서 데이터 의미 변경입니다.

📐 12강. 용량 산정은 rows보다 bytes와 조회 패턴을 함께 봅니다

ClickHouse 용량을 산정할 때 “하루 몇 건”만으로는 부족합니다. 한 행의 컬럼 수와 타입, 압축률, 보존 기간, 집계 테이블 수, replica 수, 백필 여유 공간, peak insert, peak query를 함께 봐야 합니다. 하루 1억 건이라도 컬럼이 작고 보존 기간이 짧으면 감당 가능할 수 있고, 하루 1천만 건이라도 문자열이 크고 장기 보존이면 비용이 커질 수 있습니다.

용량 산정은 저장 공간뿐 아니라 merge와 query 리소스도 포함합니다. insert가 많으면 part와 merge가 늘고, query가 많으면 CPU와 메모리가 필요합니다. 백필을 자주 한다면 평상시의 2배 이상 쓰기 부하가 생길 수 있습니다. 운영 예산에는 정상 트래픽뿐 아니라 재처리와 장애 복구 비용도 포함해야 합니다.

실무적으로는 큰 테이블부터 관측합니다. 테이블별 bytes, 컬럼별 크기, 파티션별 증가량, 쿼리별 read_bytes를 보면 비용 최적화 우선순위가 보입니다. 저장 비용이 큰 컬럼과 조회 비용이 큰 쿼리가 항상 같지는 않습니다. 저장 비용은 큰 문자열 컬럼이 만들고, 조회 비용은 자주 group by하는 컬럼 조합이 만들 수 있습니다.

🧾 13강. 데이터 계약을 문서화해야 파이프라인이 안정됩니다

ClickHouse 운영에서 데이터 계약(data contract)은 “이 테이블에 어떤 의미의 데이터가 어떤 보장으로 들어오는가”를 적은 약속입니다. 예를 들어 `raw_click_events`는 event_id가 전역 유니크인지, 중복 가능성이 있는지, event_time은 원천 발생 시각인지 수집 시각인지, delete가 올 수 있는지 명시해야 합니다.

데이터 계약이 없으면 downstream 개발자는 추정으로 쿼리를 작성합니다. 어떤 사람은 `event_time`을 발생 시각으로 보고, 다른 사람은 적재 시각으로 봅니다. 어떤 사람은 중복이 없다고 가정하고, 다른 사람은 dedup이 필요하다고 봅니다. 이런 차이가 통계 불일치로 이어집니다.

최소 문서에는 source, primary business key, version 기준, delete 처리, 중복 가능성, 지연 허용, backfill 방법, owner가 들어가야 합니다. 이는 과한 문서 작업이 아니라 장애 대응 비용을 줄이는 장치입니다. 특히 팀이 커질수록 데이터 계약의 가치가 커집니다.

🧭 14강. 마지막 판단 기준은 “복구 가능한 단순함”입니다

ClickHouse 파이프라인은 여러 고급 기능을 조합할수록 멋있어 보일 수 있습니다. 하지만 운영에서 더 중요한 것은 복구 가능한 단순함입니다. 장애가 났을 때 어떤 topic, 어떤 Flink job, 어떤 ClickHouse table, 어떤 partition을 다시 처리하면 되는지 바로 설명할 수 있어야 합니다.

복구 가능한 구조는 보통 계층이 명확합니다. raw는 원본 검증용, fact는 분석 기준 정제용, agg는 반복 집계용, serving은 API 응답용입니다. 각 계층이 어떤 입력에서 만들어지고, 같은 입력으로 다시 만들 수 있는지 알면 백필과 장애 복구가 쉬워집니다. 반대로 한 테이블에 모든 의미가 섞이면 단순해 보이지만 복구가 어려워집니다.

따라서 기능을 추가할 때마다 “이 기능은 장애 때 어떻게 되돌릴 수 있는가”를 물어야 합니다. 새 MV를 만들 때도, async insert를 켤 때도, TTL을 추가할 때도, 보존 기간을 줄일 때도 같은 질문을 적용합니다. 이 질문이 ClickHouse 시스템을 장기적으로 유지보수 가능한 방향으로 이끕니다.

✅ 6주차 정리

이번 주의 핵심은 ClickHouse를 적재, 재처리, 비용, 보존까지 포함한 운영 시스템으로 보는 것입니다. batch insert는 part 수와 merge 부담을 줄이는 기본이고, retry는 중복을 만들 수 있으므로 idempotency를 고려해야 합니다. Kafka/Flink/ClickHouse 사이의 보장 경계를 명확히 해야 합니다.

백필은 과거 데이터를 다시 넣는 행위가 아니라 기존 산출물을 안전하게 교체하는 절차입니다. TTL과 보존 정책은 비용 제어의 핵심이고, hot/cold data 구분은 SLA와 비용을 함께 설계하게 해줍니다. 최종적으로는 “틀렸을 때 다시 만들 수 있는가”가 통계 시스템의 안정성을 결정합니다.

핵심 용어

`batch insert`는 여러 행을 묶어 한 번에 넣는 방식입니다. `idempotency`는 같은 작업을 여러 번 실행해도 결과가 같게 만드는 성질입니다. `backfill`은 과거 구간 데이터를 다시 계산하거나 채우는 작업입니다. `TTL`은 데이터 보존 기간과 삭제·이동 정책입니다.
