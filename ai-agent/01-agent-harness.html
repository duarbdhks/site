🎯 먼저 정의부터 바로잡기

LLM을 한 번 호출한다고 Agent가 되지는 않습니다. 일반적인 LLM 기능은 개발자가 정한 코드 경로 안에서 텍스트를 생성하지만, Agent는 현재 상태를 보고 다음 행동을 선택하고, 도구 결과에 따라 계획을 바꾸며, 완료·실패·중단 여부까지 판단합니다. 따라서 핵심 차이는 “대화 UI가 있는가”가 아니라 모델이 실행 흐름의 일부를 통제하는가입니다.

형태실행 경로적합한 문제대표 위험
단일 LLM 호출코드가 고정분류, 추출, 요약, 초안출력 형식 오류
워크플로개발자가 단계와 분기를 정의결제, 승인, 정산처럼 순서가 중요한 업무예외 분기 증가
자율 Agent모델이 다음 단계와 도구를 선택코드 수정, 조사, 장애 진단처럼 경로가 사전에 불명확한 업무루프, 과잉 실행, 복합 오류

규완님이 일상적으로 쓰는 Claude Code나 Codex를 보면 이 차이가 선명합니다. “이 파일을 요약해줘”는 단일 호출로도 가능하지만, “이 이슈를 해결하고 테스트까지 통과시켜줘”는 코드 탐색, 파일 수정, 명령 실행, 실패 분석, 재수정이 이어집니다. 이 반복 루프가 Agent의 최소 단위입니다.

사용자 목표 ↓ 현재 상태 + 정책 + 작업 이력 ↓ LLM이 다음 행동 선택 ├─ 도구 호출 → 결과를 상태에 반영 → 다시 판단 ├─ 사용자 승인 요청 → 대기 → 재개 └─ 최종 결과 / 실패 / 안전 중단

🧩 책의 핵심 공식과 실무적 확장

원문은 Agent = LLM + Context + Tools라는 공식을 중심으로 10개 장을 전개합니다. 이 공식은 설계 지도를 잡는 데 유용하지만, 운영 가능한 시스템으로 옮기려면 모델 주변의 Harness를 함께 봐야 합니다.

🧠 모델

의도 이해, 계획, 판단을 담당합니다. 모델 교체가 가능하도록 인터페이스 뒤에 숨겨야 합니다.

👀 컨텍스트

시스템 정책, 사용자 요청, 코드와 문서, 현재 작업 상태, 도구 결과를 포함합니다.

🖐️ 도구

검색·읽기 같은 감지와 파일 수정·PR 생성 같은 실행을 담당합니다.

🛡️ Harness

허용 범위, 상태 관리, 검증, 재시도, 중단, 추적을 맡습니다.

모델은 시간이 지나면 빠르게 바뀝니다. 반면 어떤 정보를 언제 공급할지, 어떤 행동을 허용할지, 결과를 무엇으로 검증할지, 실패 후 어디서 재개할지는 제품과 조직의 고유 자산입니다. 규완님이 커리어 관점에서 노려야 할 지점도 여기에 있습니다. 프롬프트를 잘 쓰는 개발자보다 Agent가 일할 수 있는 환경을 설계하는 개발자의 역할 범위가 더 넓습니다.

🛡️ Harness의 다섯 기능

1. 컨텍스트 공급

현재 판단에 필요한 정보만 최신 상태로 제공합니다. 저장소 전체를 매번 넣는 방식은 비용과 정확도 양쪽에서 손해입니다.

2. 도구 제공

모델이 이해하기 쉬운 이름, 파라미터, 실패 의미를 가진 인터페이스를 제공합니다.

3. 제약

읽기·쓰기·네트워크·배포 권한을 기본 거부하고, 위험도에 따라 자동 허용·승인·차단을 나눕니다.

4. 검증

모델의 “완료했습니다”가 아니라 테스트, 타입 검사, 스키마, 실제 외부 상태로 성공을 확인합니다.

5. 교정과 복구

일시 오류는 재시도하고, 반복 실패는 다른 전략이나 인간에게 넘기며, 작업 상태를 보존해 이어서 실행합니다.

운영 관측

다섯 기능을 관통하는 기반입니다. 어떤 컨텍스트와 도구가 어떤 결과를 만들었는지 추적할 수 있어야 합니다.

중요한 오해
Harness는 특정 Agent 프레임워크와 같은 말이 아닙니다. 프레임워크는 구현 수단이고, Harness는 시스템이 제공해야 할 책임의 묶음입니다. 요구가 단순하면 직접 작성한 TypeScript 상태 머신이 더 유지보수하기 쉽습니다.

🪜 자율성을 올리는 의사결정 사다리

Agent를 먼저 선택하면 안 됩니다. 아래 단계에서 문제를 해결하는 가장 낮은 복잡도를 선택한 뒤, 평가 결과가 한계를 보여줄 때만 한 단계 올리는 편이 안전합니다.

결정적 코드로 가능한가?

정규식, 스키마 검증, SQL, 규칙 엔진으로 해결되면 LLM을 쓰지 않습니다.

단일 호출로 가능한가?

입력과 출력이 한 번에 정의되고 외부 상태 변경이 없다면 구조화 출력만 사용합니다.

고정 워크플로로 가능한가?

단계 순서와 승인 지점을 알 수 있다면 코드가 흐름을 통제하고 LLM은 각 단계의 판단만 맡습니다.

동적 경로가 필요한가?

다음 단계가 도구 결과에 따라 바뀌고 예외를 열거하기 어렵다면 제한된 자율 Agent를 사용합니다.

여러 Agent가 새 정보를 만드는가?

병렬 탐색, 독립 실행 검증, 권한 격리가 실질적 이득을 줄 때만 분리합니다.

예를 들어 GitHub 이슈를 수정하는 캡스톤에서 “테스트 실행 전에는 PR을 만들지 않는다”는 규칙은 코드 워크플로로 고정합니다. 반면 “어떤 파일을 먼저 조사할지”와 “실패한 테스트를 어떻게 고칠지”는 모델이 결정하도록 둡니다. 자율성과 결정성을 섞는 방식이 실무 기본값입니다.

🧱 캡스톤의 최소 상태 머신

이번 강좌 전체에서 만드는 캡스톤은 GitHub 이슈를 읽고 계획을 승인받은 뒤 코드와 테스트를 수정하고 Draft PR을 준비하는 Agent입니다. 1강에서는 SDK를 고르기 전에 상태와 전이를 먼저 정의합니다.

agent-state.ts
type RunStatus =
  | "queued"
  | "gathering_context"
  | "planning"
  | "waiting_for_approval"
  | "implementing"
  | "verifying"
  | "ready_for_pr"
  | "completed"
  | "failed"
  | "cancelled";

interface AgentRun {
  id: string;
  issueUrl: string;
  status: RunStatus;
  attempt: number;
  maxTurns: number;
  plan?: string;
  lastCheckpoint?: string;
  failure?: {
    code: "TOOL_ERROR" | "POLICY_DENIED" | "VERIFY_FAILED" | "BUDGET_EXCEEDED";
    message: string;
    retryable: boolean;
  };
}

const allowedTransitions: Record<RunStatus, RunStatus[]> = {
  queued: ["gathering_context", "cancelled"],
  gathering_context: ["planning", "failed", "cancelled"],
  planning: ["waiting_for_approval", "failed", "cancelled"],
  waiting_for_approval: ["implementing", "cancelled"],
  implementing: ["verifying", "failed", "cancelled"],
  verifying: ["implementing", "ready_for_pr", "failed", "cancelled"],
  ready_for_pr: ["completed", "failed", "cancelled"],
  completed: [],
  failed: ["gathering_context", "cancelled"],
  cancelled: []
};

export function transition(run: AgentRun, next: RunStatus): AgentRun {
  if (!allowedTransitions[run.status].includes(next)) {
    throw new Error(`Invalid transition: ${run.status} -> ${next}`);
  }
  return { ...run, status: next, lastCheckpoint: new Date().toISOString() };
}

이 코드는 모델의 사고 과정을 흉내 내지 않습니다. 대신 제품이 보장해야 할 외부 상태를 명시합니다. 프로세스가 죽어도 AgentRun을 다시 읽으면 어느 단계부터 재개해야 하는지 알 수 있고, 승인 전 구현이나 검증 전 PR 생성을 코드로 막을 수 있습니다.

중단 조건을 먼저 적으세요

  • 최대 모델 턴 수 또는 작업 시간에 도달하면 중단한다.
  • 같은 도구 오류가 3회 반복되면 자동 재시도를 멈춘다.
  • 쓰기 범위가 승인된 저장소 밖으로 벗어나면 차단한다.
  • 테스트가 실패하면 PR 단계로 이동하지 않는다.
  • 비용 예산을 넘으면 현재 체크포인트와 원인을 남기고 종료한다.

👔 커리어와 리더십 관점

개발자 단계에서는 “Agent가 동작하는가”를 보여주는 데 집중하기 쉽습니다. 팀장·CTO 트랙에서는 “어떤 업무를 Agent에 맡겨도 되는가”, “실패 비용은 누가 부담하는가”, “업그레이드 후 품질을 어떻게 증명하는가”를 결정해야 합니다. 이 질문에 답하려면 모델 API보다 소프트웨어 아키텍처, 보안, 운영, 제품 지표를 함께 다뤄야 합니다.

포트폴리오에서도 모델 이름보다 설계 결정을 설명하는 편이 강합니다. “왜 자율 Agent 대신 고정 승인 워크플로를 선택했는가”, “어떤 전이는 코드로 강제했는가”, “실패 후 어떤 상태에서 재개되는가”를 문서화하면 백엔드 역량과 리더십 판단을 동시에 보여줄 수 있습니다.
